# Smoke run — 2026-05-06

First end-to-end execution of the docsense generation pipeline against
a real LLM. Manual verification step closing out Block 1A.

**Hardware:** RTX 4070 (12 GB VRAM), display sharing the GPU
(~5.3 GB baseline usage outside this script).

**Command:**

```
HF_HOME=/mnt/e/hf-cache python scripts/smoke_generate.py --device cuda:0
```

`--device cuda:0` was needed because the default `device_map="auto"`
spilled some weights to CPU on the first attempt (limited free VRAM
post-display), causing load and inference to thrash. Forcing all-GPU
fits cleanly with ~1.9 GB margin and runs at expected speed. **Block
1B work item:** add a `max_memory` knob or a smarter default so the
typical user doesn't need this flag.

## Query

> How do I use AutoModel.from_pretrained() to load a pretrained model?

## Generated answer

> To use `AutoModel.from_pretrained()` to load a pretrained model, you
> need to call the `from_pretrained` method from the `transformers`
> library. This method can accept weights from the Hugging Face Hub or
> a local directory. Here's how you can do it:
>
> ```py
> from transformers import AutoModel
>
> # Load a pretrained model from the Hugging Face Hub or a local directory
> model = AutoModel.from_pretrained("username/modelname")
> ```
>
> If you want to specifically use `AutoModelForCausalLM`, which is a
> subclass of `AutoModel` tailored for causal language modeling tasks,
> you would do something like this:
>
> ```py
> from transformers import AutoModelForCausalLM
>
> # Load a pretrained causal language model from the Hugging Face Hub or a local directory
> model = AutoModelForCausalLM.from_pretrained("username/modelname")
> ```
>
> In both cases, replace `"username/modelname"` with the actual name
> of the model you wish to load from the Hugging Face Hub or the path
> to the local directory containing the model weights.

## Citations

**None.** Qwen 2.5 7B Instruct didn't honor the cite-by-`[N]` directive
on this run. The system prompt says "Cite sources using [N] notation
that matches the numbered context items below" and the user message
contains `[1]` through `[5]` numbered chunks, but the model produced
a clean answer without any `[N]` markers.

This is a real behavioral observation, not a bug — the wiring is
correct (the `parse_citations()` extractor is working; there's just
nothing to extract). It's exactly the kind of question Block 1B's
LLM-judge evals will systematically measure. Possible mitigations to
explore there:

- Strengthen the system prompt with an example of cited output.
- Try a Qwen-specific chat-template variant.
- Test whether instruct models in this size class (Qwen 7B, Llama 8B,
  Mistral 7B) differ in citation reliability.

## Timing

| Stage | Wall time |
|---|---:|
| Index + retriever setup | ~10 s |
| Hybrid retrieve + rerank | 6.5 s |
| Model load (Qwen 7B → NF4 on GPU) | ~163 s |
| Inference (223 tokens) | 15.3 s |
| **Total wall** | **184.7 s** |

Inference rate: **14.6 tok/sec** on contended 12 GB GPU at NF4. Solid.
The 6.5-second hybrid+rerank latency is dominated by the cross-encoder
call (~1.5 sec/chunk over 5 chunks plus overhead); the dense+sparse
retrieval itself is sub-second.

The 163 s model load is a one-time-per-session cost. Subsequent queries
in the same Python process would skip this and run at inference-speed
only — so a second query would take ~15-20 s end-to-end.

## What this confirms

1. **The pipeline composes end-to-end.** Real model, real chunks, real
   answer.
2. **Chat-template refactor (Block 1A.2) works.** Qwen produces clean
   output, no garbled tokens, no leaked system-prompt echoing, no
   prompt continuation. The `tokenizer.apply_chat_template` path is
   correct.
3. **NF4 quantization (Block 1A.3) works.** Model loads at ~5-6 GB and
   runs at typical NF4 throughput.
4. **`device_map="cuda:0"` is necessary on a 12 GB GPU shared with
   display.** Default `"auto"` is too willing to spill weights to CPU
   when free VRAM is tight, killing performance.

## What this surfaces for Block 1B and beyond

- **Citation behavior is unreliable from a single Qwen run.** Need to
  measure systematically across queries and possibly tune the prompt.
  This is precisely what `check_citations_grounded` in Block 1B will
  quantify.
- **Default `device_map` should probably not be `"auto"`** for the
  typical user. Either change the default or document the constraint
  prominently. Followup work for the smoke script and config.
- **Retrieval+rerank latency** (~6.5 s) is non-trivial. For an
  interactive demo it's tolerable; for serving (Phase 4) we'd want to
  pre-warm the cross-encoder and possibly batch queries.
